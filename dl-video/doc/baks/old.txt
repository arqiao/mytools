其中 url2md → s2w_yitang_wiki 这条是工具模块却依赖了 s2w 层的 YitangCopier。
url2md 从 s2w_yitang_wiki.YitangCopier 使用了以下方法：
  ┌───────────────────────┬────────────────────────┐
  │         方法          │          用途          │
  ├───────────────────────┼────────────────────────┤
  │ fetch_source_blocks() │ 获取一堂文档 blocks    │
  ├───────────────────────┼────────────────────────┤
  │ fetch_feishu_blocks() │ 获取飞书文档 blocks    │
  ├───────────────────────┼────────────────────────┤
  │ resolve_wiki_token()  │ wiki URL → document_id │
  ├───────────────────────┼────────────────────────┤
  │ _flatten_blocks()     │ blocks 树拍平为列表    │
  ├───────────────────────┼────────────────────────┤
  │ _block_to_md()        │ 单个 block → markdown  │
  └───────────────────────┴────────────────────────┘

  其中 resolve_wiki_token 内部又调用了 feishu_token 模块的同名函数。_flatten_blocks 和 _block_to_md 是纯数据转换，不涉及网络请求。

  本质上 url2md 需要的是 YitangCopier 的两类能力：
  1. 飞书/一堂文档抓取（fetch_*, resolve_wiki_token）
  2. blocks → markdown 转换（_flatten_blocks, _block_to_md）

  如果要消除这个反向依赖，可以把 markdown 转换部分抽成独立模块，但 fetch_* 逻辑跟 YitangCopier 的加密/认证体系深度耦合，拆分成本较高。目前这条依赖实际上是合理复用，不像之前 s2→s1w 那样有明确的层级违反，暂时可以不动。

当前的依赖结构已经比较清晰了。feishu_token 作为底层公共模块，feishu_minutes 和 feishu_auth 平级，各层调用方向合理。url2md → YitangCopier 虽然跨层但属于合理复用，拆分成本不值得。

resolve_wiki_token，在两个脚本中都有出现，他们功能没有差异。
YitangCopier.resolve_wiki_token() 就是一个薄包装，直接委托给 feishu_token.resolve_wiki_token()，只是把 self.creds 和 self.session 传进去而已。
两者功能完全一致，不需要改名。这种 "实例方法包装模块函数" 的模式是合理的——让 YitangCopier 的调用方不需要关心 creds/session 的传递细节。
